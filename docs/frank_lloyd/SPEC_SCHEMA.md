# Frank Lloyd Spec Schema
*Status: DRAFT — design pass only.*  
*Last updated: 2026-04-11*

---

## Overview

A Frank Lloyd spec is a **YAML frontmatter + Markdown body** document. The frontmatter contains all structured fields that Frank Lloyd and the approval workflow consume programmatically. The markdown body contains the human-readable context that the operator needs to make an informed review decision.

Both sections are required. A spec without structured fields cannot be validated. A spec without human context cannot be reviewed meaningfully.

---

## Format

```
---
<structured YAML fields>
---

## Context
...

## What changes
...

## What does not change
...

## Integration steps
...

## Operator review checklist
...
```

---

## Structured fields (YAML frontmatter)

### 1. Core identity — required for all build types

```yaml
id: string              # e.g. "BUILD-001" — auto-generated by Frank Lloyd
title: string           # short human-readable name for this build
created: ISO8601        # e.g. "2026-04-11T14:00:00Z"
requested_by: string    # "operator" | agent name
build_type: enum        # see Build Types below
risk_level: enum        # see Risk Levels below
status: enum            # see Status Values below
autonomy_stage_required: int  # minimum Frank Lloyd stage that can execute this (1|2|3|4)
```

**Build types:**

| Value | Meaning |
|---|---|
| `new_house` | Creating a brand-new house from scratch |
| `new_service` | Creating a new backstage operating service |
| `modification` | Modifying any existing file in any house or service |
| `platform_capability` | Adding a new cross-cutting platform capability |
| `clone` | Duplicating an existing house as a new instance |
| `docs_only` | Documentation-only change — no code |

**Risk levels:**

| Value | Conditions |
|---|---|
| `low` | Docs only; OR new file in an isolated location with no integration requirements and no existing file changes |
| `medium` | New file that requires route registration or integration; new backstage service from template |
| `high` | Any modification to an existing file; any new house build; any change touching existing route logic or data schemas |
| `critical` | Any change to `app/main.py`, `scripts/ctl.sh`, `app/routes/neighborhood.py`, or any runtime infrastructure |

Note: a build can have multiple risk levels if it touches multiple files. The **highest** risk level of any individual change governs the overall build risk level.

**Status values:**

| Value | Meaning |
|---|---|
| `draft` | Spec in progress — not ready for review |
| `spec_approved` | Operator approved the spec — Frank Lloyd may proceed to staging |
| `in_build` | Frank Lloyd is generating artifacts |
| `staged` | Artifacts are in staging area, Sentinel has run |
| `in_review` | Operator is reviewing staged artifacts |
| `approved` | Operator approved promotion |
| `promoted` | Artifacts moved to live paths |
| `rejected` | Operator rejected — no promotion |
| `aborted` | Aborted during build due to abort condition |

---

### 2. State — required for all build types

```yaml
current_state: string   # what exists right now (be specific — file paths, endpoint names, behaviors)
target_state: string    # what should exist after this build (be specific)
```

---

### 3. Files — required for all build types except docs_only

```yaml
files:
  create:
    - path: string              # relative to repo root, e.g. "app/routes/event_query.py"
      description: string       # what this file does
      template_source: string   # optional: which existing file is this modeled on
  modify:
    - path: string
      description: string       # what specific change is being made
      requires_explicit_approval: bool   # always true for existing files — Frank Lloyd sets this
  delete:
    - path: string
      reason: string
```

Any entry in `modify` or `delete` automatically triggers `requires_explicit_approval: true`. Frank Lloyd must not set this to `false` for existing files.

---

### 4. Approvals — required for all build types

```yaml
approvals:
  before_build:
    - string    # what the operator must confirm before Frank Lloyd starts generating
  before_promotion:
    - string    # what the operator must confirm before artifacts go live
  never_automate:
    - string    # things that must never be automated regardless of Frank Lloyd's maturity stage
```

The `never_automate` list must name anything that falls under the permanent off-limits rules from BUILDER_SPEC.md §12. Frank Lloyd should populate this list honestly — do not omit items to make the build look safer.

---

### 5. Reuse — required for all build types except docs_only

```yaml
reuse:
  capabilities_used:
    - name: string        # e.g. "LMHelper"
      module: string      # e.g. "app/cost_warden"
      how: string         # short description of how it's being reused
  new_platform_capabilities_needed:
    - name: string
      why: string         # why no existing capability covers this need
```

If `new_platform_capabilities_needed` is non-empty, this build implicitly spawns future platform work. Flag it explicitly so the operator can track it.

---

### 6. Sentinel — required for all build types except docs_only

```yaml
sentinel:
  required: bool          # should be true for any build that touches code
  scope: smoke | auto | full
  files_under_test:
    - string              # which files Sentinel should check
  expected_outcome: string  # e.g. "safe verdict on all files"
```

If `sentinel.required: false` for a code build, Frank Lloyd must explain why in the markdown Context section. The default is `true`.

---

### 7. Success criteria — required for all build types

```yaml
success_criteria:
  - criterion: string
    verifiable_by: string   # how to verify this criterion was met
    automated: bool         # can this be checked automatically or does it require human verification
```

---

### 8. Rollback and abort — required for all build types

```yaml
abort_if:
  - string    # conditions under which Frank Lloyd should stop and not produce artifacts

rollback:
  after_promotion: string    # what to do if the promoted build needs to be reversed
  staging_cleanup: string    # what to clean up if the build is rejected or aborted before promotion
```

---

### 9. House eligibility — required for new_house and clone only

```yaml
eligibility:
  durable_role:
    assessment: string
    verdict: pass | questionable | fail
  measurable_outcomes:
    assessment: string
    verdict: pass | questionable | fail
  some_autonomy:
    assessment: string
    verdict: pass | questionable | fail
  standalone_value:
    assessment: string
    verdict: pass | questionable | fail
  real_workflow:
    assessment: string
    verdict: pass | questionable | fail
  overall_verdict: house | backstage_service | not_ready
  notes: string
```

If `overall_verdict` is not `house`, Frank Lloyd must not proceed to build a house. It should present the assessment and stop.

---

### 10. Architecture — required for new_house, new_service, platform_capability, clone

```yaml
architecture:
  layer: experience | executive_control | specialist_house | operating_services | platform
  violates_layer_discipline: bool
  requires_new_routes: bool
  modifies_existing_house: bool
  modifies_runtime_infrastructure: bool
  requires_neighborhood_ui_changes: bool
  notes: string
```

If `violates_layer_discipline: true`, Frank Lloyd must stop and explain the violation. Frank Lloyd may not rationalize away a layer violation in order to proceed.

## Required vs optional fields by build type

| Section | new_house | new_service | modification | platform_capability | clone | docs_only |
|---|---|---|---|---|---|---|
| Core identity | required | required | required | required | required | required |
| State | required | required | required | required | required | required |
| Files | required | required | required | required | required | omit |
| Approvals | required | required | required | required | required | required |
| Reuse | required | required | required | required | required | omit |
| Sentinel | required | required | required | required | required | omit |
| Success criteria | required | required | required | required | required | required |
| Rollback/abort | required | required | required | required | required | partial* |
| Eligibility | required | omit | omit | omit | required | omit |
| Architecture | required | required | omit | required | required | omit |

*For docs_only: only `staging_cleanup` is relevant.

## Markdown body sections

All sections are required unless marked optional.

### `## Context`
Why this build is being requested. What problem it solves. What triggered the request. Any background the operator needs before reviewing.

### `## What changes`
Detailed description of what will be created or modified. Should be concrete enough that an operator could predict what the finished artifact looks like before seeing it.

### `## What does not change`
Explicit scope boundary. Name things that might seem related but are out of scope for this build. This prevents scope creep during review.

### `## Integration steps`
What needs to happen after the artifacts are promoted: which route include to add in main.py, which data files to initialize, which neighborhood panel to update. If integration requires touching main.py or neighborhood.py, name the exact change.

### `## Operator review checklist`
A concrete list of things the operator should verify before approving promotion. Should be checkable, not vague. Example: "run `GET /events?limit=5` and confirm it returns the five most recent events from the event log."

## Example spec: Event log query endpoint

```markdown
---
id: "BUILD-001"
title: "Event log query endpoint — GET /events"
created: "2026-04-11T14:00:00Z"
requested_by: "operator"
build_type: platform_capability
risk_level: critical
status: draft
autonomy_stage_required: 2

current_state: >
  No query endpoint exists for the event log. The only way to read events is
  via Peter's /peter/status (which applies a fixed filter) or by reading
  data/event_log.jsonl directly. No filtering by agent, severity, or time range
  is available via the API.

target_state: >
  A new GET /events endpoint at app/routes/event_query.py that accepts query
  parameters: agent (optional), severity (optional), since (ISO8601, optional),
  limit (int, default 20, max 200). Returns events newest-first. Registered in
  app/main.py. Uses read_recent_events() from observability/event_log.py.

files:
  create:
    - path: "app/routes/event_query.py"
      description: "New read-only FastAPI router exposing GET /events with query filters"
      template_source: "app/routes/custodian.py"
  modify:
    - path: "app/main.py"
      description: "Add one include_router() call for the new event_query router"
      requires_explicit_approval: true

approvals:
  before_build:
    - "Operator confirms spec is accurate and scope is correct"
  before_promotion:
    - "Operator reviews diff for app/routes/event_query.py"
    - "Operator reviews the specific include_router line added to app/main.py"
    - "Sentinel verdict: safe or operator-accepted review"
    - "Operator runs GET /events?limit=3 against staged artifacts and confirms response shape"
  never_automate:
    - "The include_router() addition to app/main.py must be reviewed by operator before promotion"

reuse:
  capabilities_used:
    - name: "read_recent_events"
      module: "observability/event_log"
      how: "Call directly; wrap with parameter parsing for agent/severity/since filters"
  new_platform_capabilities_needed: []

sentinel:
  required: true
  scope: smoke
  files_under_test:
    - "app/routes/event_query.py"
  expected_outcome: "safe verdict — new isolated route file with no side effects"

success_criteria:
  - criterion: "GET /events returns events newest-first"
    verifiable_by: "Call GET /events?limit=5; confirm timestamps are descending"
    automated: true
  - criterion: "agent filter works correctly"
    verifiable_by: "Call GET /events?agent=belfort; confirm all returned events have source=belfort"
    automated: true
  - criterion: "limit is respected and capped at 200"
    verifiable_by: "Call GET /events?limit=500; confirm response contains at most 200 events"
    automated: true
  - criterion: "Missing event log file returns empty list, not error"
    verifiable_by: "Sentinel smoke test covers this path"
    automated: true

abort_if:
  - "event_log.py read_recent_events signature has changed — spec would need updating"
  - "Sentinel returns not_ready for event_query.py"

rollback:
  after_promotion: >
    Remove app/routes/event_query.py. Remove the include_router line from
    app/main.py. No data is affected — the endpoint is read-only.
  staging_cleanup: "Delete staging/frank_lloyd/BUILD-001/"

architecture:
  layer: platform
  violates_layer_discipline: false
  requires_new_routes: true
  modifies_existing_house: false
  modifies_runtime_infrastructure: false
  requires_neighborhood_ui_changes: false
  notes: "Read-only query layer over existing event_log platform capability. No state mutation."
---

## Context

The event log (`data/event_log.jsonl`) is the primary audit trail for the system. Currently, there is no API endpoint that exposes it with filtering. Peter's `/peter/status` reads a small fixed slice. The dev dashboard may read it directly. Frank Lloyd's own build log will eventually append to a similar log. Having a proper query endpoint makes the audit trail accessible without file access and sets the pattern for future log query capabilities.

## What changes

A new file `app/routes/event_query.py` is created. It exposes one endpoint:

```
GET /events
  ?agent=<string>         filter by source agent (optional)
  ?severity=<string>      filter by severity level (optional)
  ?since=<ISO8601>        return only events after this timestamp (optional)
  ?limit=<int>            max results, default 20, max 200
```

Returns: `{"events": [...], "count": int, "filtered_by": {...}}`

One line is added to `app/main.py` to register the router.

## What does not change

- `observability/event_log.py` — used as-is, no modifications
- `data/event_log.jsonl` — read-only access only
- Peter's existing status endpoint — not replaced or modified
- No neighborhood UI changes
- No other route files

## Integration steps

1. Promote `staging/frank_lloyd/BUILD-001/artifacts/app/routes/event_query.py` to `app/routes/event_query.py`
2. Add to `app/main.py` in the router registration block:
   ```python
   from app.routes.event_query import router as event_query_router
   app.include_router(event_query_router)
   ```
3. Restart backend: `./scripts/ctl.sh restart`
4. Verify: `curl http://localhost:8001/events?limit=3`

## Operator review checklist

- [ ] `event_query.py` uses `read_recent_events()` from `observability/event_log` — no direct file reads
- [ ] No write operations in event_query.py (GET only)
- [ ] Limit is enforced and capped at 200
- [ ] No new data files created
- [ ] The include_router line in main.py is the only change to main.py
- [ ] Sentinel verdict: safe
- [ ] `GET /events?limit=3` returns valid JSON with events list
```
